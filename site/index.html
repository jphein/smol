02 — The Game

Block Digger

Gravity pulls you down. Walk the surface, dig blocks into your inventory, then place them back to build. It's Minecraft distilled to its purest loop and squeezed onto 2,880 pixels — the whole world fits in a screen the size of a postage stamp.

  • Procedural world: sky, grass, dirt, stone
  • Real gravity + auto step-up walking
  • Dig / place with a live inventory counter
  • Renders to a 1-bit framebuffer at ~60 fps
Why not real Minecraft?

The honest math.

Bedrock needs ~2–4 GB of RAM and a GPU. smol has 400 KB and a 1-bit screen — off by roughly 10,000×. So instead of faking it, Block Digger keeps the soul of Minecraft (dig, gather, build) at a scale the hardware loves. The mockup on the left is the actual render logic, running in your browser.

03 — Controls

A Stadia pad over Bluetooth.

The board has just two buttons — one of them is RESET — so a real controller matters. A Stadia controller flashed to Bluetooth mode is a standard BLE gamepad, and the C3 is BLE-only, so it's a perfect match. The Bluepad32 library handles pairing; the game reads the D-pad and face buttons.

10 — Influences

Standing on shoulders.

smol didn't appear from nowhere. Its ESP-NOW → MQTT → Home Assistant plumbing is commodity — a well-trodden pattern many have built. What's actually novel is the whole: a no_std Rust game-console mesh on $3 boards, a deliberate single-radio burst time-share, and a retained-payload → mesh-rebroadcast downlink to display-only leaves. Below are the projects that shaped it, grouped by the thread each runs in — credited honestly, with what smol borrows and where it diverges. A few handed over portable ideas; most gave lineage, identity and nerve rather than code.

◇ Mesh & routing
  • MESHTASTICThe mesh ethos smol admires — multi-hop, self-healing, friendly per-node names, an MQTT bridge to the world. smol's realm-sigil board names and its #13/#14 roadmap trace straight back here; different PHY — ESP-NOW on 2.4 GHz, not LoRa. ↗
  • PAINLESSMESHA self-organizing WiFi mesh where every node is both STA and AP. The road smol didn't take — it chose ESP-NOW for lower power, lower latency and tighter game timing. The honest counterfactual. ↗
  • ALTHEAAn OpenWrt-based decentralized, incentivized mesh ISP — self-organizing routed mesh, per-link WireGuard tunnels, Babel routing, IPv6, bandwidth settled by micropayments on its own L1 chain. smol borrows the self-organizing-mesh + per-link-encryption ethos (its lightweight analog is ESP-NOW LMK keys, #12) and the routed-multi-hop idea (Babel is prior art for #13); it leaves behind the blockchain settlement and full IPv6-over-mesh — node-ids are smol's lightweight addressing. The most different influence — an L3 incentivized ISP vs smol's L2 game mesh — architectural inspiration, not a template. ↗
  • ZHNETWORK · aZHOLTIKOVA routed, multi-hop ESP-NOW mesh library that bridges to MQTT + Home Assistant autodiscovery. Direct prior art for true mesh routing — and ahead of smol here: smol does single-hop relay today and lists routed multi-hop as #13 roadmap, not shipped. ↗
  • ESP-NOW → MQTT → HA · LATTICThe exact ESP-NOW → MQTT → Home Assistant gateway pattern smol independently arrived at — direct prior art for smol's relay/collector bridge, credited plainly. ↗
◇ Local-first ESP ↔ HA
  • BATTERY-EMULATORESP32 firmware (LilyGo, C++) that emulates a BMS to bridge repurposed EV battery packs to home solar inverters over CAN bus, with its own web UI. The one influence smol is actually wired to: in JP's own stack, smol subscribes to its BE/* MQTT battery telemetry — a live interop neighbor, not just an idea. It bridges CAN↔inverter on the mains side; smol is a battery-powered ESP-NOW mesh display/relay — different job, same MQTT-native, local-first, HA-first philosophy. ↗
  • TASMOTAOpen-source, MQTT-first, local-first alternative firmware for ESP smart-home devices — a rich cmnd/stat/tele topic scheme, runtime config over MQTT (no reflash), an on-device web UI, a rules engine, robust OTA, HA MQTT discovery. smol's closest genre-cousin: it borrows the runtime-config-over-MQTT / no-reflash ethos (exactly what the node manager does), the MQTT topic conventions, and the OTA + HA-discovery + local-first values (smol just firewalled off a cloud-phoning logger). Differs in form — Tasmota is C++/Arduino on always-on-WiFi mains devices with a web UI; smol is no_std Rust, burst/battery, single-radio, so the control surface relocates to HA. Prior art for local-MQTT-ESP→HA that smol independently lands in a game-console/mesh form. ↗
  • WLEDThe gold standard for Home Assistant polish — device-grouped typed entities, auto-discovery, its own ESP-NOW sync, a slick on-device web UI. smol borrows the entity-structure lesson; it can't host an on-device UI (the burst radio is deaf most of the time), so its node manager lives in HA instead. ↗
  • ESPHOMEThe YAML→firmware ecosystem and its always-connected native API. smol stays MQTT-native on purpose — the persistent native API fights the single-radio burst model, and no no_std-Rust ESPHome exists to build on anyway. ↗
  • ESP_WIFI_REPEATER · MARTIN-GERProof that an ESP is a real router — concurrent STA+AP, NAT, a console, MQTT, ACLs. Inspires smol's gateway-as-network-citizen role and shows concurrent multi-role radio is achievable. ↗
  • ESP3DESP8266/ESP32 firmware that bridges a 3D-printer or CNC serial port onto WiFi with a web UI — serial-over-WiFi, telnet, even Home Assistant notifications. smol borrows the ESP-as-WiFi-bridge-with-web-control appliance ethos: make a dumb serial device networked. Differs — ESP3D is one device ↔ one browser; smol is a multi-node mesh whose UI is the OLED + HA, not a per-node web page. ↗
  • ESP-LINK · JEELABSAn ESP8266 WiFi↔serial bridge that can remotely reflash an attached AVR/ARM MCU over WiFi (an stk500v1 programmer fed by HTTP hex upload), with a web console. smol borrows the gateway-bridge ethos and cites it as prior art for firmware-over-WiFi alongside OTA (#6). The honest distinction: esp-link flashes a separate attached chip — it's the programmer, not the target — while smol's OTA self-updates the ESP itself into an inactive flash slot. Same spirit, different mechanism. ↗
◇ Cheap-ESP audacity — smol's soul
  • ESP32 MARAUDERA WiFi/BLE multi-tool on a cheap ESP32 with a tiny touchscreen and a menu-driven handheld UI. smol borrows the form-factor genre — cheap ESP + tiny screen + menu navigation is exactly smol's physical shape and UX — plus a radio-manipulation kinship (both bend the 2.4 GHz radio to their will). Differs — smol is a cooperative mesh node, not a pentest tool, and it manipulates its own radio for ESP-NOW coexistence, not other people's networks. ↗
  • LEAFMINERA FOSS Bitcoin solo-miner for the ESP8266/ESP32 (NerdMiner-family, built on NerdSHA256plus) — openly a learning toy, since actually mining a block is astronomically unlikely. smol borrows the soul: a $5 chip doing something gloriously improbable for the joy and the learning of it — the same spirit as a shared-world snake MMO running on a clock. Differs — smol is a useful mesh (telemetry, time, display), not a lottery ticket. ↗
  • ATOMIC14Five single-button games (Breakout, Micro Racer, Defender, Jump Run, Heli Cave) built for this exact ESP32-C3 + 0.42″ OLED — and now flashed onto smol. The closest thing to a direct ancestor: living proof this precise board is a game console. Squarely in smol's play lineage. ↗
◇ Remote-management UX
  • PIKVMAn open-source IP-KVM on a Raspberry Pi — a pure-HTML5 web UI for remote keyboard/video/mouse, power control and BIOS access, with no client install. smol borrows the remote-management-appliance ethos and its clean web-control UX: manage the physical thing from anywhere, through a browser. Aspirational for smol's HA-side control surface and the future #21 node manager. Differs — PiKVM is a Linux Pi driving one other computer; smol is a fleet of bare-metal ESP32 mesh nodes managed via HA/MQTT + an OLED. ↗
◇ Lean real-time embedded
  • GRBLReal-time CNC motion-control firmware in tightly optimized C for an Arduino-class AVR — a G-code planner that looks 16 moves ahead, emits up to 30 kHz of jitter-free step pulses, and honors real-time override commands mid-stream. smol borrows the lean-real-time-firmware craft — deterministic behavior squeezed from a tiny MCU — and a loose design echo: GRBL interleaves out-of-band real-time commands with a streamed buffer, roughly the shape of smol draining ESP-NOW RX between MQTT poll cycles. Differs — GRBL is single-purpose motion with no radio or mesh; the echo is lineage, not a code steal. ↗

◇ BLE presence (the #22 basis) — smol's beacon-tracking idea would build on Bermuda (HACS trilateration that reuses existing bluetooth_proxy anchors) and ESPresense (dedicated MQTT anchors). Prior art and reference, not code smol borrows.

11 — Pocket Watch

We designed smol a pocket watch.

No printable pocket-watch case existed for this board, so an agent generated one from scratch — a parametric round shell in the classic orientation: the chain bail and a removable crown-style USB-C cover stack at the top, with the OLED window, BOOT/RESET plunger bores and two clear-PLA LED light-pipes below. Inside: pockets for the board + a 502030 LiPo, a TP4056 charge slot, and a press-fit back lid. Three printable parts — body, lid, crown — and the OLED is rotated 180° in firmware so it reads upright when hung. These are live renders of the actual STL.
